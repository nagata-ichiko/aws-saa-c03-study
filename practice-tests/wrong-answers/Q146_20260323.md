# Q146 - S3 保存データの暗号化方式

- 日付: 2026-03-23
- 分野: Domain 1（セキュアなアーキテクチャ）
- 結果: ❌ 不正解
- 自分の回答: A, C, F
- 正解: A, B, E

## 問題文

ある企業が Amazon S3 にデータを保存しています。同社のセキュリティポリシーでは、保存データの暗号化が義務付けられています。これを実現する方法として適切なものはどれでしょうか？（3つ選択）

A. AWS Key Management Service 管理キーを使用した Amazon S3 のサーバーサイド暗号化を利用する。(SSE-KMS)
B. 顧客提供のキーを使用した Amazon S3 のサーバーサイド暗号化を利用する。(SSE-C)
C. EC2 キーペアを使用した Amazon S3 のサーバーサイド暗号化を利用する。
D. Amazon S3 バケットポリシーで保存データへのアクセスを制限する。
E. 独自のマスターキーを使ってクライアント側でデータを暗号化してから Amazon S3 に取り込む。
F. Amazon S3 への転送中のデータを SSL で暗号化する。

## 解説

### 正解: A, B, E

**A. SSE-KMS（正解）**: AWS KMS が管理するキーで S3 がサーバーサイドで暗号化する方式。キーの管理・監査・ローテーションを AWS KMS が担い、保存データ（data at rest）を暗号化する。

**B. SSE-C（正解）**: 顧客が自分で用意した暗号化キーを S3 に渡し、S3 がそのキーでサーバーサイド暗号化を行う方式。キーの管理は顧客側の責任だが、保存データの暗号化として有効。

**E. クライアントサイド暗号化（正解）**: データを S3 にアップロードする前にクライアント側で暗号化する方式。独自のマスターキーで暗号化してから S3 に保存するため、保存データは暗号化された状態になる。

### 不正解: C, D, F

**C. EC2 キーペアによるサーバーサイド暗号化（不正解）**: EC2 キーペアは EC2 インスタンスへの SSH 接続用であり、S3 の暗号化メカニズムとしては存在しない。S3 のサーバーサイド暗号化は SSE-S3、SSE-KMS、SSE-C の3種類のみ。

**D. バケットポリシーによるアクセス制限（不正解）**: バケットポリシーはアクセス制御（認可）の仕組みであり、暗号化ではない。アクセスを制限しても、データ自体は暗号化されていない。

**F. SSL による転送中の暗号化（不正解）**: SSL/TLS はデータの転送中（data in transit）の暗号化であり、保存データ（data at rest）の暗号化ではない。問題で問われているのは「保存データの暗号化」である。

### S3 暗号化方式のまとめ

| 方式 | キー管理 | 暗号化の場所 | 保存データ暗号化 |
| :--- | :--- | :--- | :--- |
| SSE-S3 | AWS が完全管理 | サーバーサイド | ✅ |
| SSE-KMS | AWS KMS で管理 | サーバーサイド | ✅ |
| SSE-C | 顧客が管理 | サーバーサイド | ✅ |
| クライアントサイド暗号化 | 顧客が管理 | クライアントサイド | ✅ |
| SSL/TLS | - | 転送経路 | ❌（転送中のみ） |
