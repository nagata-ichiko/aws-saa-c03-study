# Q158 - マルチリージョン低レイテンシー設計（DynamoDB vs ElastiCache）

- 日付: 2026-03-23
- 分野: Domain 3（高性能アーキテクチャの設計）
- 結果: ❌ 不正解
- 自分の回答: B
- 正解: A

## 問題文

複数国のユーザーをサポートするソーシャルメディアサイト。複数リージョンで高可用性設計。最もレイテンシーに敏感なのはユーザーの好み読み取り（パーソナライズ・広告選択）。どのオプションが要件を満たすか？

### 選択肢

- A. S3+CloudFront配信、Route 53レイテンシーベースルーティング、各リージョンのローカルDynamoDBテーブル、SQSで変更伝播
- B. S3 Copy APIで複数リージョンコピー、S3+CloudFront+ELB配信、各リージョンのElastiCacheクラスター、SNS通知で変更伝播
- C. S3 Copy APIで複数リージョンコピー、Route 53レイテンシーベースルーティング、DynamoDBテーブル（単一）、SQSで変更伝播
- D. S3+CloudFront配信、各リージョンのELB、各リージョンのElastiCacheクラスター、SWFで中央DBから伝播

## 解説

### なぜ A が正解か

- **CloudFront**: S3 のコンテンツを自動的にエッジキャッシュで配信。S3 Copy API で手動コピーする必要がない
- **Route 53 レイテンシーベースルーティング**: ユーザーを最寄りのリージョンに自動ルーティング → レイテンシー最小化
- **各リージョンのローカル DynamoDB テーブル**: ミリ秒単位の低レイテンシー読み取り。ユーザーの好みデータの永続的な保存に最適
- **SQS で変更の非同期伝播**: リージョン間のデータ同期を結果整合性で実現（ユーザー好みデータは即時整合性不要）

### なぜ B が不正解か

- S3 Copy API は手動コピーで非効率（CloudFront なら自動キャッシュ）
- ElastiCache はインメモリで高速だが、永続的なユーザー好みデータの保存には DynamoDB の方が適切（永続性・マルチリージョン対応）
- **Route 53 レイテンシーベースルーティングがない** → ユーザーを最寄りリージョンに振り分ける仕組みが欠けている

### なぜ C が不正解か

- DynamoDB テーブルが単一リージョン → 遠いリージョンのユーザーはレイテンシーが高くなる
- S3 Copy API は不要（CloudFront で十分）

### なぜ D が不正解か

- SWF（Simple Workflow Service）は複雑すぎる。データ伝播には SQS や SNS で十分
- Route 53 レイテンシーベースルーティングがない

### 学習ポイント: DAX vs ElastiCache vs DynamoDB ローカルテーブル

- **DynamoDB ローカルテーブル（各リージョン）**: マルチリージョンでの永続的データ保存。グローバルテーブル機能で自動レプリケーションも可能
- **DAX**: DynamoDB 専用のインメモリキャッシュ。マイクロ秒単位の読み取りが必要な場合に追加
- **ElastiCache**: 汎用キャッシュ（Redis/Memcached）。アプリ側でキャッシュロジック実装が必要
