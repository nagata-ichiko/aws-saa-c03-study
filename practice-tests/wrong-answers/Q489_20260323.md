# Q489 - 不正解記録

## 問題情報
- **問題ID**: 489
- **日付**: 2026-03-23
- **トピック**: DynamoDB コスト最適化、SQS バッファリング
- **分野**: Domain 4（コストを最適化したアーキテクチャの設計）

## 問題
モバイル睡眠追跡アプリのバックエンドコスト最適化の問題（2つ選択）

## 選択肢
- A. 毎日新しい Amazon DynamoDB テーブルを作成し、データが Amazon S3 に保存された後に前日のテーブルを削除する。
- B. Amazon S3 に保存された JSON ファイルの代わりに、モバイルアプリが Amazon DynamoDB に直接アクセスするようにする。
- C. Amazon SQS キューを導入して Amazon DynamoDB テーブルへの書き込みをバッファリングし、プロビジョニングされた書き込みスループットを削減する。
- D. Amazon ElastiCache を導入して Amazon DynamoDB テーブルからの読み取りをキャッシュし、プロビジョニングされた読み取りスループットを削減する。
- E. Amazon DynamoDB と Amazon S3 の両方を置き換えて、Amazon Redshift クラスターに直接データを書き込む。

## 正解: A, C
## 自分の回答: A, B

## 間違えた理由
- B を選択してしまった: S3 を廃止して DynamoDB に直接アクセスさせることがコスト削減になると誤解した。実際には S3 の方がストレージコストが安価であり、DynamoDB への直接アクセスはコスト増加につながる。
- C を選択しなかった: SQS によるバッファリングが DynamoDB の書き込みスループットを平準化し、プロビジョニングコストを削減できることを見落とした。

## 学ぶべきポイント
- SQS バッファリングパターン: トラフィックスパイクの平準化によりプロビジョニング容量を削減できる
- DynamoDB のコスト最適化: 不要なテーブルの削除、書き込みバッファリング、適切なキャパシティ設定
- S3 vs DynamoDB のストレージコスト: 大量データの保存には S3 の方が圧倒的にコスト効率が良い
