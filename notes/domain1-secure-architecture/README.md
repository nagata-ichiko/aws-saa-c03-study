## 📝 学習ノート

### IAM の基礎

IAM（Identity and Access Management）は AWS リソースへのアクセスを制御するサービスです。
試験では、**最小権限の原則**（Principle of Least Privilege）に基づいたポリシー設計が頻出です。

**ポリシー評価ロジック（重要）**

1. デフォルトはすべて **Deny**
2. 明示的な **Allow** があれば許可
3. 明示的な **Deny** があれば、Allow があっても **必ず拒否**

**IAM ロール vs IAM ユーザー**

| 項目 | IAM ユーザー | IAM ロール |
| :--- | :--- | :--- |
| 認証情報 | 長期（パスワード・アクセスキー） | 一時的（STS トークン） |
| 主な用途 | 人間のオペレーター | AWS サービス・外部 ID プロバイダー |
| ベストプラクティス | MFA を有効化、アクセスキーは極力使わない | EC2 インスタンスプロファイルなどで活用 |

### ネットワークセキュリティの階層防御

VPC 内のリソースを保護するためには、セキュリティグループとネットワーク ACL を組み合わせて使用します。

| 特徴 | セキュリティグループ | ネットワーク ACL (NACL) |
| :--- | :--- | :--- |
| 適用範囲 | インスタンスレベル (ENI) | サブネットレベル |
| ステート | ステートフル (戻りのトラフィックは自動許可) | ステートレス (戻りのトラフィックも明示的な許可が必要) |
| ルール | 許可ルールのみ | 許可ルールと拒否ルールの両方 |
| 評価順序 | すべてのルールを評価 | 番号の若い順に評価 |

### AWS セキュリティサービスの守備範囲（Shield / WAF / CloudFront）

外部攻撃に対する防御は **「何の攻撃を、どのレイヤーで防ぐか」** で整理するのが最も効果的です。

**防御レイヤーの全体像:**

```
インターネット
  │
  ▼
┌─────────────────────────┐
│  AWS Shield              │ ← L3/L4: DDoS 攻撃を吸収
│  (Standard / Advanced)   │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│  Amazon CloudFront       │ ← エッジで配信 + 地理的制限（国単位ブロック）
│  (CDN)                   │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│  AWS WAF                 │ ← L7: SQL インジェクション、XSS、
│  (Web Application FW)    │      IP ブロック、レートリミット
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│  ALB → EC2              │ ← アプリケーション
└─────────────────────────┘
```

**AWS Shield — DDoS 防御専用の「盾」**

| 項目 | Shield Standard | Shield Advanced |
| :--- | :--- | :--- |
| 料金 | 無料（自動適用） | 月額 $3,000 |
| 防御対象 | L3/L4 の DDoS | L3/L4/L7 の高度な DDoS |
| 追加機能 | なし | DDoS 対応チーム (DRT)、コスト保護、WAF 料金込み |

Shield でできないこと: SQL インジェクション対策、地理的制限、特定 IP の個別ブロック。
→ Shield は「大量トラフィックによる攻撃（DDoS）を受け止める盾」とだけ覚える！

**AWS WAF — Web アプリの「門番」（L7）**

WAF でできること:
- SQL インジェクション / XSS のブロック
- 特定 IP アドレスのブロック
- リクエストレートの制限（レートベースルール）
- 特定の国からのブロック（IP 地理情報で）

WAF を関連付けできるサービス: CloudFront, ALB, API Gateway, AppSync

**CloudFront Geo Restriction — 国単位のアクセス制御**

- 国単位でアクセスを許可/拒否（ホワイトリスト or ブラックリスト）
- 設定がシンプル（コンソールで数クリック）

**試験での判断フローチャート:**
- 「DDoS」→ Shield
- 「SQL インジェクション / XSS」→ WAF
- 「特定の国をブロック」→ CloudFront Geo Restriction
- 「国ブロック + アプリ攻撃防御」→ CloudFront + WAF（組み合わせ）

### S3 暗号化方式の使い分け

| | SSE-S3 | SSE-KMS | SSE-C | クライアントサイド |
| :--- | :--- | :--- | :--- | :--- |
| 誰がキーを管理？ | AWS が完全管理 | AWS KMS で自社管理 | 自社が毎回提供 | 自社で完全管理 |
| キーの種類 | S3 マネージドキー | KMS キー（CMK） | 自社のキー | 自社のキー |
| 自動ローテーション | 制御不可 | 年1回自動 | 自前実装 | 自前実装 |
| CloudTrail で監査 | 不可 | 誰がいつキーを使ったか記録 | 不可 | 不可 |
| キーポリシーで制御 | 不可 | 誰がキーを使えるか制御可能 | 不可 | 不可 |
| 運用負荷 | 最小 | 低い | 高い | 最も高い |

**試験での判断フローチャート:**
- 「暗号化すればOK、細かい要件なし」→ SSE-S3
- 「キーを自社管理 / ローテーション / 監査」→ SSE-KMS (CMK)
- 「AWS にキーを預けたくない」→ SSE-C
- 「AWS にデータを暗号化前の状態で渡したくない」→ クライアントサイド暗号化

**覚え方:** KMS = Key Management Service = キーの管理サービス → キー管理の要件 = KMS！
