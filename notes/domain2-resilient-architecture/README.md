## 📝 学習ノート

### 高可用性の設計パターン

**マルチ AZ 構成** は AWS における高可用性の基本です。
同一リージョン内の複数の AZ にリソースを分散させることで、単一障害点（SPOF）を排除します。

**RDS Multi-AZ vs Read Replica**

| 項目 | Multi-AZ | Read Replica |
| :--- | :--- | :--- |
| 目的 | 可用性向上（フェイルオーバー） | 読み取りパフォーマンス向上 |
| レプリケーション | 同期 | 非同期 |
| 読み取り可否 | 不可（スタンバイ） | 可能 |
| 自動フェイルオーバー | あり（約1〜2分） | なし |

### 疎結合アーキテクチャ

**SQS** を使った非同期処理は、コンポーネント間の依存を減らし、スケーラビリティを高める典型的なパターンです。

| SQS キュータイプ | 特徴 | ユースケース |
| :--- | :--- | :--- |
| 標準キュー | 高スループット、順序保証なし、1回以上の配信 | 大量メッセージ処理 |
| FIFO キュー | 順序保証あり、1回のみ配信、スループット制限あり | 金融取引、注文処理 |

### ディザスタリカバリ (DR) 戦略

RTO (目標復旧時間) と RPO (目標復旧時点) に応じて、適切な DR 戦略を選択します。

1. **バックアップと復元**: 最も安価だが、復旧に時間がかかる (RTO/RPO が数時間〜数日)
2. **パイロットライト**: コアサービスのみを最小限で稼働させ、障害時にスケールアウトする
3. **ウォームスタンバイ**: 縮小版の環境を常に稼働させておく (RTO/RPO が数分)
4. **マルチサイトアクティブ/アクティブ**: 複数のリージョンでフルスケール稼働 (RTO/RPO がほぼゼロ、最も高価)
